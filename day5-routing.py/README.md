## 자주 나온 질문
- 아직 잘 안 잡히는 부분
위의 내용이 맞나요?? ==> 요청해서..흠..입력한 내용을 분석하고, 어떤 경로(노드)로 보낼건지 결정한다 가 너 나을거 같아요.

라우팅 라우팅 워크플로는 입력을 처리한 후 컨텍스트별 작업으로 전달합니다. 이를 통해 복잡한 작업을 위한 특수화된 흐름을 정의할 수 있습니다. 예를 들어, 제품 관련 질문에 답변하기 위해 구축된 워크플로는 먼저 질문 유형을 처리한 후, 가격 책정, 환불, 반품 등과 같은 특정 프로세스로 요청을 라우팅할 수 있습니다.

- gemini-2.0 모델을 사용했을 때 GPU 성능이 많이 요구된다고 하는데 2.5와 차이가 많은 건지 모르겠습니다. ==> 유저입장에서는 보편적으로 모델 버전이 높으면 최신 모델이라 더 똑똑한 답변을 기대하긴 합니다.

- 요약이 맞는지 모르겠음. ==> 키워드 여러개를 미리 세팅해서 if문처럼 특정키워드로 들어오면 특정 llm을 실행시킨다..?
어떻게 보면 비슷하죠. 근데 특정 llm을 실행시킨다는 말보다는 입력을 기준으로 어떤 노드로 갈지 결정한다에 가깝습니다. 거기 llm이 없을수도 있죠.

- 이게 프롬프트 엔지니어링인가? ==> 프롬프트 엔지니어링은 LLM이 원하는 방향으로 응답하도록 언어적으로 유도하는 기법입니다. 라우팅은 입력을 기준으로 다음에 실행할 노드를 결정하는 구조라고 보시면 됩니다. 프롬프트 엔지니어링과 라우팅은 중점이 다르다고 볼 수 있을거 같아요.

- Parallel Execution (병렬 처리) 노드들 ==> 병렬 구조로 만들었다면, 결과를 한 곳에 모아주는 aggregator같은걸 만들어주면 좋을 것 같아요.
